# Prism 0.6.0 Design Cycle

## Goal

Bundle the two API gaps surfaced by arbor migration waves 1-5 into a single Prism 0.6.0 release, with one optional third gap if `/analyze` surfaces a recurring pattern across the four shipped `*-prism` routes (documents, payslips, clients, employees).

## Inputs

### Queued candidates

1. **prism#24 — `FilterBar` molecule** (G-2 workaround promotion).
   - Origin: wave-2 G-2 documented workaround.
   - Consumers hit: `documents-prism` (wave-3), `my-payslips-prism` (wave-3), `clients-prism` (wave-4), `employees-prism` (wave-5, second DataTable).
   - Pattern: search input + chip-row OR dropdown + view-mode toggle outside DataTable; adapter rebuilt on `(filterTuple)` change; ~60 LOC of inline filter CSS per page; 1-2 `useState` hooks per page.
   - Proposal: `FilterBar` molecule owns layout + state via `useFilterBarState<TFilters>()`; `DataTable.filtering` accepts FilterBar adapter so engine handles delta internally without consumer-level adapter recreation.

2. **prism#25 — Synthetic computed columns** (relax `ColumnDef.field`).
   - Origin: wave-5 `/employees-prism` Profile-completeness column.
   - Pattern: derived %, aggregate badges, computed monetary totals, cross-field labels — all currently rejected by `field: string & keyof T`.
   - Proposal: relax `field: string`; if `keyof T`, engine reads `row[field]`; otherwise `undefined` and `sortable` MUST be false; runtime check for sortable+synthetic combo.

### Investigation scope

Sweep workspace `fe-codegen-platform/journal/0026-0029` + four arbor `*-prism` routes + wave-4 plan § "Out-of-scope" for any third recurring gap. Hypothesis: bundling a likely-third gap now avoids 0.6.0/0.6.1 ping-pong. Cap: if no third gap recurs across ≥2 waves, ship 0.6.0 with two features and resume migrations.

### Out of scope

- Implementation. `/analyze` produces failure-point analysis + design proposal; `/todos` plans the shards; `/implement` writes the code.
- Wave-6 migration. Resumes after 0.6.0 ships.
- Other Prism repos (Flutter, Tauri). Web engine only — these gaps are React/TypeScript-specific.

## Constraints

- **API surface promotion only**: no breaking changes to existing exported atoms/molecules unless `/analyze` proves the existing surface is structurally wrong. `ColumnDef.field` relaxation IS such a case (compile-time TS narrowing changes); evaluate runtime back-compat at `/analyze`.
- **Tier 2 storybook coverage** for every new molecule per `docs/specs/05-engine-specifications.md` Filter Engine section that doesn't yet exist.
- **Spec-first**: `docs/specs/05-engine-specifications.md` § Filter Engine + § DataTable updates land BEFORE any code. Spec is the contract.
- **Migration recipe update**: confirm that the `data ?? []` useMemo wrap pattern (R19 from wave-5) still applies post-0.6.0; codify in `docs/specs/05-engine-specifications.md` § DataTable § Consumer Patterns or in COC `co-setup/12-consumer-migration-pattern.md` if/when the migration recipe gets promoted.
