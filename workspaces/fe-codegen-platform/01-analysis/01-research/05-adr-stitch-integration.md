# ADR-001: Stitch Integration Strategy

## Status: Proposed

## Context

Google Stitch (stitch.withgoogle.com) is a design generation tool that produces high-fidelity UI mockups and HTML/Tailwind CSS from text, image, or voice input. It uses dual-tier Gemini models (Flash for speed, Pro for fidelity) and introduces the DESIGN.md protocol -- a machine-readable Markdown specification for design systems (colors, typography, spacing, components).

Stitch provides an MCP server (`@_davideast/stitch-mcp`) with 7 tools: `build_site`, `get_screen_code`, `get_screen_image`, `generate_screen_from_text`, `list_projects`, `list_screens`, `extract_design_context`.

The COC frontend stack currently has zero design generation capability. The uiux-designer agent can analyze and critique designs but cannot generate them. Every design decision is made by the code generation model (Claude Opus) during implementation, which conflates design and code concerns.

### The Problem Being Solved

Three specific problems Stitch could address:

1. **Design-code conflation**: Claude Opus makes aesthetic decisions (color selection, layout composition, spacing rhythm) simultaneously with code decisions (component structure, state management, framework patterns). This produces "AI slop" because the model optimizes for code correctness at the expense of design quality.

2. **No visual iteration**: Text-based code generation cannot see its output. Stitch generates visual mockups that can be iterated on before any code is written, separating design validation from implementation.

3. **No design system extraction**: When working with an existing application or brand, there is no automated way to extract a design system specification. Stitch's `extract_design_context` tool could fill this gap.

### Constraints

- Stitch is a Google Labs product with no guaranteed stability or long-term availability.
- The MCP server is a community package (`@_davideast/stitch-mcp`), not an official Google integration.
- Stitch generates HTML/Tailwind CSS only -- not React components, not Flutter widgets, not TypeScript.
- The Kailash SDK is a Terrene Foundation project with strict independence rules. No structural dependency on any commercial entity's product.
- The Foundation independence rules prohibit designing against any proprietary product or creating coupling to proprietary systems.

## Decision

**Adopt the DESIGN.md protocol as a standard. Integrate Stitch as an optional, non-required design generation tool.**

Specifically:

### 1. Adopt DESIGN.md as the Design Specification Protocol (Independent of Stitch)

Define a machine-readable design specification format based on the DESIGN.md concept. This format will be:
- A Markdown document that specifies colors, typography, spacing, shadows, border radii, component variants, and layout patterns
- The single source of truth for design decisions in any frontend project
- Consumable by both human designers and AI agents
- Framework-agnostic (maps to CSS variables, Tailwind config, and Flutter theme constants)
- Authored manually, generated by Stitch, extracted from existing designs, or produced by any future tool

The protocol is owned by the Kailash COC system, not by Stitch. Stitch is one possible generator; manual authoring is equally valid.

### 2. Integrate Stitch MCP as an Optional Accelerator

Add Stitch MCP integration to the COC frontend workflow as a non-required step:

```
WITHOUT STITCH (manual path):
  Brief -> Author DESIGN.md manually -> Generate tokens -> Compose from primitives -> Validate

WITH STITCH (accelerated path):
  Brief -> Stitch generates mockup -> Extract DESIGN.md from mockup -> Generate tokens -> Compose from primitives -> Validate
```

Both paths converge at the same point: a DESIGN.md spec that feeds into the token generation pipeline. Stitch accelerates the early stages but is not required.

### 3. Do NOT Depend on Stitch for Any Critical Path

No COC command, agent, or skill will REQUIRE Stitch to function. Every Stitch-dependent step will have a manual fallback. If Stitch is unavailable (network, API change, discontinuation), the workflow continues at reduced speed but full functionality.

### 4. Do NOT Adopt Stitch's Generated Code

Stitch generates HTML/Tailwind CSS. This is useful as a visual reference but should NOT be used as production code. The COC workflow generates production code from DESIGN.md specs using the primitives library. Stitch's code output is a mockup, not an implementation.

Rationale: Stitch code does not use our component primitives, does not follow our architecture patterns (elements/ structure, one API call per component), and does not include framework-specific features (React hooks, Next.js server components, Flutter state management).

## Consequences

### Positive

1. **Design-code separation**: DESIGN.md creates a clean boundary between design decisions and implementation decisions. Design is validated before code is written.

2. **Tool independence**: By owning the DESIGN.md protocol, the system is not coupled to any specific tool. Stitch, Figma plugins, manual authoring, or future AI tools can all produce valid DESIGN.md files.

3. **Cross-session persistence**: DESIGN.md is a file that persists across sessions. Session 2 reads the DESIGN.md from session 1 and produces consistent output without re-deriving design decisions.

4. **Framework agnosticism**: DESIGN.md specifies values (colors, spacing, typography) that map to any framework. One DESIGN.md produces both React/Tailwind tokens and Flutter theme constants.

5. **Visual iteration capability**: When Stitch is available, the uiux-designer agent gains design generation capability -- it can create mockups, iterate on them, and extract validated specs before any code is written.

### Negative

1. **DESIGN.md protocol must be defined**: This is new work. The protocol does not exist as a formal spec in the Kailash ecosystem. It must be designed, documented, and validated.

2. **Token generation pipeline**: A tool/script must be built to convert DESIGN.md to framework-specific token files (CSS variables, Tailwind config, Flutter constants). This is the T-01 gap from the gap analysis.

3. **Stitch MCP reliability**: The community MCP server may have bugs, incomplete coverage, or break with Stitch updates. Integration testing is required.

4. **Two workflow paths**: Supporting both "with Stitch" and "without Stitch" paths adds complexity to the guidance and command structure.

5. **Google Labs risk**: Stitch may be discontinued. The mitigation (tool independence via DESIGN.md protocol) reduces this risk but cannot eliminate it entirely -- teams that become accustomed to visual iteration will lose that capability.

## Alternatives Considered

### Alternative A: Full Stitch Integration (Rejected)

**Description**: Make Stitch the primary design tool. All frontend projects start with Stitch, use Stitch's generated code as the base, and iterate in Stitch.

**Pros**: Maximum design quality from specialized AI. Visual iteration built in. MCP integration allows agent-driven design.

**Cons**: Hard dependency on a Google Labs product (violates Foundation independence). Generated code is HTML/Tailwind, not React/Flutter (requires translation layer). No cross-session persistence of design decisions without a separate protocol. Single vendor lock-in.

**Rejection reason**: Creates a structural dependency on a commercial entity's product, which violates the Foundation's independence charter. Also, Stitch's code output is not suitable as production code.

### Alternative B: No Stitch Integration (Considered but not chosen)

**Description**: Ignore Stitch entirely. Build the DESIGN.md protocol, token pipeline, and primitives without any design generation tool.

**Pros**: Zero external dependency. Simpler system. No risk of Stitch discontinuation.

**Cons**: No visual iteration capability. Design decisions are still made by the code generation model during implementation. The design-code conflation problem remains unsolved for the "design generation" phase. Manual DESIGN.md authoring is slow and requires design expertise.

**Why not chosen**: Stitch provides genuine value for the design generation step. Rejecting it entirely because of dependency risk is overly conservative when the risk can be mitigated by making it optional.

### Alternative C: Figma Integration Instead (Considered but deferred)

**Description**: Integrate with Figma (via Dev Mode API or plugins) instead of Stitch for design generation and spec extraction.

**Pros**: Figma is the industry standard for design. More stable than Google Labs. Rich ecosystem of plugins and integrations.

**Cons**: Figma is a proprietary product with paid plans. Requires a Figma account and potentially a license. The API is more complex than Stitch's MCP. Figma does not generate designs from text -- it is a design tool, not a design AI.

**Why deferred**: Figma integration is complementary, not competing. A future phase could add Figma as another DESIGN.md source (extract design tokens from Figma files). This does not conflict with the current decision.

### Alternative D: Build Custom Design Generation (Rejected)

**Description**: Build a custom design generation system using Claude/Gemini to produce visual mockups directly.

**Pros**: Full control. No external dependency. Can be tailored to enterprise requirements.

**Cons**: Massive development effort (design generation is a hard AI problem). Would duplicate what Stitch already does. Does not align with the "no code in this repo" directive for loom.

**Rejection reason**: Building a design generation AI is a multi-year project. Leveraging existing tools (Stitch as optional) is the rational choice.

## Implementation Plan

### Phase 1: DESIGN.md Protocol Definition (1 session)
- Define the DESIGN.md spec format (colors, typography, spacing, shadows, border radii, component variants, layout patterns)
- Create 2-3 example DESIGN.md files (enterprise professional, startup modern, minimal)
- Document the spec as a COC skill
- Write token generation script (DESIGN.md to CSS variables + Tailwind config + Flutter constants)

### Phase 2: Manual Workflow Validation (1 session)
- Author DESIGN.md manually for a real project
- Generate tokens from DESIGN.md
- Compose a page from primitives + generated tokens
- Validate with /i-audit: score target 35+

### Phase 3: Stitch MCP Integration (1 session)
- Install and evaluate the Stitch MCP server
- Build a /stitch command (or extend /design) that invokes Stitch for mockup generation
- Build the extraction pipeline: Stitch mockup -> DESIGN.md
- Test the accelerated workflow on a real project
- Validate: same /i-audit score but faster time-to-production

### Phase 4: Feedback Loop (ongoing)
- Monitor Stitch availability and API stability
- Evaluate alternative design generation tools as they emerge
- Evolve the DESIGN.md protocol based on project needs

## Decision Criteria for Future Review

This ADR should be revisited if:
1. Stitch is discontinued or paywalled
2. A superior open-source design generation tool emerges
3. The DESIGN.md protocol proves insufficient for complex design systems
4. Claude or another code model gains native visual generation capability (rendering previews from code)
