# Red Team: Arbor Wave 2 Migration

**Date:** 2026-04-14
**Reviewer:** reviewer agent (code review pass)
**Parallel reviewer:** security-reviewer (separate doc)
**Scope:** M-01 calculators-prism + M-02 my-payslips-prism + M-03 documents-prism + M-04 Prism 0.2.0 remediation

## Summary

The M-04 Prism 0.2.0 remediation is structurally correct: D1 (`ServerDataSource.fetchData` wiring) is properly wired, tested via a new Tier 1 regression test (`server-data-source-wiring.test.ts`, 4 cases), and the DataTableAdapter design in § 5.1.1 is correctly marked "PROPOSED" with no accidental implementation. Zero-Tolerance Rule 2 is clean across the wave (no TODOs, stubs, or mock data in production paths). Spec compliance is strong — the Form and DataTable engine specs match the shipped code.

The wave has **two material defects** that undermine the stated user-visible win: (1) the F4 `classNames` override was added to Prism but **zero** arbor migration pages consume it, so the off-brand Form fields (M-01 BLOCKING-5, the only regression explicitly flagged in that migration) still ship; and (2) the `act()` warnings surfaced by M-04 and deferred are a **Zero-Tolerance Rule 1 violation** — tests pass but emit 8 distinct warning categories (33 `act()` call sites in the suite), and the disposition "shard discipline" is not one of the permitted exceptions. A regression-parity gap was also found in `/documents-prism` list view (rows are no longer clickable links) and a semantic divergence in the Notice Period calculator (bespoke has a subtle double-accounting bug that prism silently corrects).

Verdict: **Issues Found.** HIGH findings must land before wave-3 migrations start; wave-2 is functional but carries a user-visible regression that contradicts the M-04 CHANGELOG's claim.
