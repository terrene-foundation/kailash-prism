### [DRIFT] [05b] DataTable virtualScroll default contradicts spec

- **Spec**: `docs/specs/05b-data-table-engine.md:180` — virtual scrolling MUST activate when row count > 100 (auto-activation, no flag required).
- **Source**: `web/src/engines/data-table/data-table-root.tsx:82` — `virtualScroll = false` (default disabled; consumer must opt-in via prop).
- **Why this matters**: The spec promises auto-activation as a baseline UX guarantee; production code makes it opt-in. Either the spec is aspirational and needs correction (per `rules/specs-authority.md` MUST Rule 6 — log deviation in the spec file), or the production default is wrong. Per `rules/specs-authority.md`, deviations from spec require explicit acknowledgment in the spec file — and there is no such acknowledgment in `05b`.
- **Recommended next step**: User decides — change source default to auto-activate above 100 rows, OR update spec to reflect intentional opt-in default (cite the rationale in spec body).

### [DRIFT] [05b] Spec still documents removed `ServerDataSource` interface

- **Spec**: `docs/specs/05b-data-table-engine.md:112-114` — documents `ServerDataSource<T>` as a current contract: "Since 0.2.0: wired end-to-end; `fetchData` is invoked on every state transition".
- **Source**: `web/src/engines/data-table/types.ts:153` (header comment): "`ServerDataSource<T>` was removed in 0.3.0. Consumers migrating from..." — the symbol was removed; spec was not updated.
- **Why this matters**: Per `rules/specs-authority.md` MUST Rule 5 ("spec files are updated at first instance"), removing a public symbol MUST update the spec in the same action. The spec currently advertises a contract the code does not honor — direct contract violation visible to consumers reading docs.
- **Recommended next step**: Update `docs/specs/05b` lines 110-129 to remove `ServerDataSource<T>` references AND add a § "Removed" entry citing the 0.3.0 cut.

### [COVERAGE GAP] [05e] Form engine performance contracts not benchmarked

- **Spec**: `docs/specs/05e-form-engine.md:237-244` — Field render < 2ms, Validation < 10ms, Conditional visibility < 5ms.
- **Source**: `web/src/engines/form/__tests__/form-engine.wiring.test.tsx` exists (paired-operation round-trip verified) — type/contract coverage present.
- **Gap**: No `*.bench.ts` or `performance.now()`-instrumented tests exercising the < N ms claims. Performance contracts are unverifiable.
- **Why this matters**: Performance MUSTs without benchmark coverage are aspirational claims that drift silently. Either codify a benchmark harness OR downgrade to "target" / "goal" language per `rules/spec-accuracy.md`.

### [COVERAGE GAP] [05h] Layout zone unmount-vs-hide untested

- **Spec**: `docs/specs/05h-layout-engine.md:68-71` — zones marked `responsive.{breakpoint}: hidden` MUST NOT render content (unmount, not `visibility:hidden`).
- **Source**: `web/src/engines/layout.tsx:50-54` — `LayoutEngineConfig` exported, wiring test exists in `web/src/engines/layout/__tests__/layout-engine.wiring.test.tsx`.
- **Gap**: No test asserting unrendered zones are unmounted (e.g. `expect(screen.queryByText(...)).toBeNull()`) vs. visibly hidden. Could pass spec by either route.
- **Why this matters**: The MUST distinguishes runtime-cost behaviors. Without a test asserting unmount, a regression to `visibility:hidden` would ship green.

### [COVERAGE GAP] [05i] Theme contrast validator unverified (also: phantom citation risk)

- **Spec**: `docs/specs/05i-theme-engine.md:76` — "Contrast compliance: constraint validator MUST verify WCAG 2.1 AA ratios."
- **Source**: No file matching `*contrast*` or `*wcag*` validator under `web/src/engines/theme/` or `compiler/src/`. No automated contrast assertion in any token test.
- **Gap**: Validator either does not exist OR exists undocumented. Per `rules/spec-accuracy.md` Rule 1, the spec references "the constraint validator" without citing which module/tool implements it — phantom citation risk.
- **Why this matters**: Accessibility claims that are not tested are claims that will fail in the field. Either implement the validator (and cite it in the spec) OR remove the MUST clause until implementation lands.
