## 0.6.0 Additions

### § Filter Engine — `FilterBar` Molecule

**Introduced 0.6.0.** Companion molecule to `DataTable`, sitting above it in a typical list-page layout. Promotes the G-2 documented workaround from a per-consumer pattern to a composable molecule. Implementation lives in `web/src/molecules/filter-bar/`; machine-readable contract in `specs/components/filter-bar.yaml`.

#### Molecule shape

```typescript
import { FilterBar, useFilterBarState } from "@kailash/prism-web";

const state = useFilterBarState<Client, { sector: string }>({
  data: clients,
  initial: { sector: "All" },
  derive: { sector: (rows) => deriveSectors(rows) },
});

<FilterBar
  search={state.search}
  onSearchChange={state.setSearch}
  searchPlaceholder="Search clients…"
  dimensions={[
    { key: "sector", label: "Sector", options: state.options.sector,
      value: state.filters.sector,
      onChange: (v) => state.setFilter("sector", v),
      shape: "dropdown" },
  ]}
  viewMode={{ active: view, options: ["grid", "list"], onChange: setView }}
/>
```

#### What `useFilterBarState` absorbs

The hook owns three concerns the consumer would otherwise hand-roll:

1. **Search + raw filter state** — `search`, `setSearch`, `filters` (raw), `setFilter`.
2. **Derive options from data** — when a `derive[K]` callback is supplied, `options[K]` is computed by calling it with `data`, deduping, sorting, and prepending `"All"` if absent.
3. **Effective-filter fallback** — `state.filters[K]` returns the *effective* value: if the raw value is not in `options[K]`, it falls back to `input.initial[K]`. This protects against orphaned selection when the last item of a category is deleted (default-on per `workspaces/prism-0.6.0/journal/0002-DISCOVERY-*.md`).

#### Why the engine is NOT extended

`DataTableAdapter.filterDimensions` was declared in `data-table.yaml` v0.2.2 and reserved for 0.4.0. As of 0.6.0 it remains **vapourware** — declared in the type, never wired, zero call sites in the engine. Per `rules/orphan-detection.md` Rule 1, interface methods without wired consumers are not shipped. The molecule-level state hook is sufficient for every observed consumer; engine-side faceted-filter wiring waits for a real consumer demand.

#### Three canonical consumer shapes

`web/src/molecules/filter-bar/__stories__/filter-bar.stories.tsx` covers all three observed arbor patterns:

- **search-only** — minimal shape, matches `my-payslips-prism`.
- **search + chip-row category** — matches `documents-prism`.
- **search + dropdown sector + view-mode toggle** — matches `clients-prism` / `employees-prism`.

A composite story `filter-bar-with-data-table.stories.tsx` exercises FilterBar above a DataTable with adapter rebuild on filter delta.

### § DataTable — `ColumnDef.field` Contract Relaxation (0.5.0 → 0.6.0)

**Changed 0.6.0.** The `ColumnDef.field` type narrowing is relaxed to support synthetic computed columns.

#### Before (0.5.0)

```typescript
interface ColumnDef<T extends DataTableRow> {
  field: string & keyof T;
  render?: (value: T[keyof T] | undefined, row: T) => ReactNode;
  // ...
}
```

#### After (0.6.0)

```typescript
interface ColumnDef<T extends DataTableRow> {
  field: string;          // dual contract — see below
  render?: (value: unknown, row: T) => ReactNode;
  // ...
}
```

#### Dual contract for `field`

- **`field` is a key of `T`** — the engine reads `row[field]` and passes the value as `value` to `render`. Sorting / filtering / global-search use this lookup. This is the typed path; existing keyed columns continue to work without modification.
- **`field` is a synthetic id** — any string the engine has never seen as a key on `T` (e.g. `"_profile"`, `"actions"`). No value lookup; `render` receives `undefined` as `value` and must compute its display from `row` alone. **`sortable` MUST be `false`** for synthetic columns (the engine throws at first non-empty render if a synthetic field is marked sortable — see `assertNoSyntheticSortable` in `use-data-table.ts`).

Synthetic columns are how consumers express derived totals, profile-completeness bars, action menus, and other display columns that don't correspond to a single field on the row.

#### Migration: `render` callback

Render callbacks that destructure the typed `value` parameter (e.g. `(value, row) => value.toLocaleString()`) will see TS errors at consumer recompile. Three migration shapes:

- **Already-coalesced**: `(value, row) => value ?? "—"` → no change needed.
- **Destructured / typed access**: cast — `(value, row) => (value as number).toLocaleString()` — or guard.
- **Synthetic-field new pattern**: `(_value, row) => <ProfileBar pct={profileCompleteness(row)}/>` — works as-is.

#### Runtime guard

Synthetic field with `sortable: true` produces semantically wrong sort (every row's `row[synthetic]` is `undefined`, comparator returns `0` or stable lexical "undefined" order — meaningless). The engine now throws at first non-empty render with:

```
column "${field}" has sortable: true but is a synthetic field
(no row[field] lookup). Synthetic columns MUST set sortable: false.
To sort by a derived value, pre-compute it into the row data before
passing to DataTable.
```

The throw fires once per `(columns identity, first non-empty data)` mount cycle. Empty data defers the check.

#### Latent type-guard fixes

The relaxation surfaced 7 previously-unguarded `row[col.field]` reads in `data-table-body.tsx` and `data-table-mobile.tsx`. The analysis sweep had marked these "already safe" via `String(value ?? '')` coalesce in the surrounding render expression, but the value-extraction itself was a typed-only access that broke when `field` widened. Fixed in the same shard per `autonomous-execution.md` Rule 4 — body got an inline cast, mobile lifted to `const rowRecord = row as Record<string, unknown>` once per render. See `workspaces/prism-0.6.0/journal/0004-DISCOVERY-*.md` Finding 1.
